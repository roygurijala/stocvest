The user reported a regression: "I am actively working on the app and I got the message that my session has expired. How can that happen. it should happen only on inactivity." The audit traced the root cause to a three-part architectural flaw — `infra/cognito.tf` never overrode Cognito's default `id_token_validity` (60 min), `frontend/lib/auth/cognito.ts:signIn()` captured only the `IdToken` from the Cognito response and silently discarded the `RefreshToken`, and `SessionExpiryWatcher` scheduled a hard `setTimeout(() => markSessionExpired("expired"), exp)` at the JWT's `exp` claim regardless of activity. The fix introduces a proper sliding-session contract: a continuously-active user with a valid Cognito refresh token stays signed in indefinitely (default 30-day refresh-token lifetime), while a truly-inactive user is signed out only when the refresh token expires or is revoked. Implementation lands across six surfaces — (1) server-side capture of `RefreshToken` in `cognito.ts:signIn()` + `respondToNewPasswordChallenge()`, plus a new `refreshIdToken(refreshToken)` that calls Cognito with `AuthFlow: REFRESH_TOKEN_AUTH`; (2) `session-cookies.ts` now writes a third httpOnly cookie `stocvest_refresh_token` (30-day `maxAge`) alongside the existing auth + mirror cookies, and `clearSessionTokenCookies()` clears all three; new `lib/auth/refresh-token-cookie.ts` for server-only reads; login + new-password actions thread the refresh token through; (3) new `POST /api/auth/refresh` BFF route — reads the refresh cookie, calls Cognito, rewrites ID + mirror cookies, returns `200 { exp }` on success or clears all cookies + returns `401` on failure; (4) new client `lib/auth/refresh-session.ts` with single-flight coalescing (parallel callers all await the same in-flight POST so we never trigger a refresh storm) and a 5-second post-failure cooldown (so a hard-expired refresh token doesn't get hammered); (5) `SessionExpiryWatcher` rebuilt to schedule a refresh ~2 min before `exp` instead of firing the expired event, re-evaluates on `visibilitychange → visible` + `focus`, only marks expired if the refresh call returns `false`; (6) `browserApiFetch` now does refresh-and-retry on 401 (one retry with the freshly-rewritten mirror cookie); only marks expired if the refresh fails OR the retry also returns 401.
