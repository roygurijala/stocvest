Root cause was a three-part architecture flaw: (a) `infra/cognito.tf` never sets `id_token_validity` so Cognito's default of 60 minutes was in effect, (b) `frontend/lib/auth/cognito.ts:signIn()` captured only the `IdToken` and silently discarded the `RefreshToken` from the same Cognito response, and (c) `SessionExpiryWatcher` decoded the JWT `exp` claim and scheduled `setTimeout(() => markSessionExpired("expired"), exp)` — firing at exactly the wall-clock moment regardless of how active the user was. The fix implements a proper sliding-session contract: a continuously-active user with a valid Cognito refresh token stays signed in indefinitely (default 30-day refresh-token lifetime), while a truly-inactive user is signed out only when the refresh token expires or is revoked. **Implementation:** (1) **server-side** — `frontend/lib/auth/cognito.ts` now captures `RefreshToken` from both `signIn()` and `respondToNewPasswordChallenge()` and exposes a new `refreshIdToken(refreshToken)` that calls Cognito with `AuthFlow: REFRESH_TOKEN_AUTH`. `frontend/lib/auth/session-cookies.ts` now writes a third httpOnly cookie `stocvest_refresh_token` (`maxAge = 30 * 24 * 60 * 60` matching Cognito's default refresh-token validity) alongside the existing `stocvest_auth_token` (httpOnly) and `stocvest_ws_token` (non-httpOnly mirror), and clears all three on `clearSessionTokenCookies()`. `frontend/lib/auth/refresh-token-cookie.ts` (new) exposes `refreshTokenCookieName()` + `readRefreshTokenCookie()` for server-only consumers. The login (`frontend/app/login/actions.ts`) and new-password (`frontend/app/new-password/actions.ts`) actions thread the refresh token through to the cookie setter. (2) **BFF route** — `frontend/app/api/auth/refresh/route.ts` (new) is a `POST` handler that reads the refresh-token cookie, calls Cognito `refreshIdToken()`, rewrites the ID + mirror cookies via `setSessionTokenCookiesFromIdToken(idToken)` (no `refreshToken` argument — Cognito does not rotate refresh tokens on `REFRESH_TOKEN_AUTH` in our pool config, so the existing 30-day cookie stays valid), and returns `200 { exp }` on success. On any failure (no cookie, Cognito throws, Cognito returns 200 without `IdToken`) it clears all three cookies and returns `401 { error: "refresh_failed" | "no_refresh_token" }` so the client falls back to the calm `SessionExpiredBanner`. Same-origin only, no request body read, runtime "nodejs" because the AWS SDK requires Node APIs. (3) **client refresh helper** — `frontend/lib/auth/refresh-session.ts` (new) exposes `refreshSession()` with **single-flight coalescing** (parallel callers — e.g. five concurrent 401s on a busy dashboard — all await the same in-flight POST so we never kick off a refresh storm; the in-flight slot clears on `setTimeout(0)` so callers that resolved against the same promise can still observe the result) and a **5-second failure cooldown** (after a refresh fails, subsequent calls short-circuit to `false` without hitting the BFF, because the calm banner is already the right UX surface and we don't want to hammer Cognito with a refresh-storm). (4) **proactive watcher** — `frontend/components/auth/session-expiry-watcher.tsx` is rebuilt: instead of firing `markSessionExpired()` at `exp`, it schedules a `setTimeout` to fire `refreshSession()` `REFRESH_LEAD_MS = 2 minutes` *before* `exp`, re-evaluates on `visibilitychange → visible` + `focus`, and only marks expired if the refresh call returns `false`. On a successful refresh it re-runs `evaluate()` to reschedule from the new `exp` automatically. (5) **reactive 401 retry** — `frontend/lib/api/browser-api-fetch.ts` now attempts `refreshSession()` on 401 and retries the original request **once** with the freshly-rewritten mirror cookie; only marks expired if the refresh itself fails OR the retry also returns 401. **403 is unchanged** — STOCVEST uses 403 for product-rule denials (PDT lockout, margin, paid-tier gating) where the user IS authenticated, and those must never trigger refresh-or-expire (locked in by `tests/browser-api-fetch-refresh.test.ts` and `tests/surface-auth-error.test.ts`). (6) **shared fallback** — `frontend/lib/auth/surface-auth-error.ts` (used by the six client fetch helpers that don't go through `browserApiFetch` — `fetch-symbol-bars`, `fetch-symbol-snapshot`, `fetch-macro-context`, `ticker-news-panel`, `public-signals`, the STOCVEST Assistant) is now async and attempts a refresh first; on success it returns `true` without marking expired (the caller's next interaction picks up the new cookie automatically) and only marks expired if the refresh fails. The six callers were updated to `await` it (the assistant uses the return value to short-circuit) or `void` it (the five fire-and-forget helpers). **No Terraform changes** — `ALLOW_REFRESH_TOKEN_AUTH` is already in `infra/cognito.tf` `explicit_auth_flows` for both app clients, and the 30-day refresh-token validity is Cognito's default which matches the cookie `maxAge`. **Dev mock unchanged** — `loginAsDevUser` mints an unsigned JWT with `exp = now + 365d` and never goes through Cognito, so it has no refresh token; the refresh route gracefully returns `401 { error: "no_refresh_token" }` if a dev-mock session ever hits it, and the watcher never schedules a refresh because the dev `exp` is a year out. **Behavior contract:** continuously-active user → stays signed in indefinitely up to the 30-day refresh-token lifetime, never sees the banner. Inactive for the full 60-minute ID-token lifetime → next interaction refreshes silently, user does not notice. ## 4. Legal compliance (product & API)

- STOCVEST is **not** a registered investment adviser; frame as **signal intelligence** and data tooling.
- **UI:** Prefer signal summary, signal strength, reference levels, signal parameters—not “verdict,” “recommendation,” “you should,” or “confidence” in **signal** context.
- **`SignalDisclaimerChip`** on signal cards; **`GlobalDisclaimer`** last in `<body>` in root layout.
- **Performance page:** Directional accuracy only; not dollar P&L as “signal performance.”
- **Serialized API:** Use `signal_summary` / `signal_strength` (and related names) where applicable; include `disclaimer`: *Signal data for informational purposes only. Not investment advice.*
- **Logging / persistence:** Sanitize user-supplied free text before Dynamo (`text_sanitize`); keep CloudWatch logs free of unnecessary PII (`log_privacy`). Server-side identity for journal and protected routes comes from the **JWT**, not from client-supplied `user_id` body fields.

## 7. Immutable contracts

Do **not** change without discussion and coordinated updates:

- **`BrokerAdapter`** — eight methods, types, exceptions (`docs/API_CONTRACTS.md` §1).  
- **HTTP paths** — versioned under `/v1/`; breaking changes need `/v2/` or migration.  
- **DynamoDB table names** — `Users`, `BrokerConnections`, `Watchlists`, `Alerts`, `Orders`, **`SignalHistory`**, **`TradeJournal`**, **`PDTState`**, **`AuditEvents`** (HTTP audit items; `pk`/`sk` per `audit_store.py`), etc.  
- **Secrets path** — `/stocvest/{userId}/brokers/{brokerId}`.  
- **Canonical types** — `stocvest/data/models.py` (no raw Polygon dicts in core logic).  
- **Snapshot + intraday + VIX invariants** (verified; do not re‑discover in prompts):
  1. **`Snapshot` model** — Pydantic `Snapshot` in `stocvest/data/models.py` (not a separate parser class). Raw Polygon ticker dicts map only through `PolygonClient._parse_snapshot()` → `Snapshot`. **Do not** add a `SnapshotParser` class, **do not** read raw Polygon keys in layer/analyzer code, **do** use `Snapshot` field names everywhere: `symbol`, `last_trade_price`, `last_trade_size`, `last_quote_bid`, `last_quote_ask`, `day_open`, `day_high`, `day_low`, `day_close`, `day_volume`, `day_vwap`, `prev_close`, `change`, `change_percent`, `pre_market_price`, `pre_market_change_percent`, `after_hours_price`, `after_hours_change_percent`, `market_status`, `company_name`, `prev_day_volume`. Layer analyzers take **`Snapshot`** instances (or derived inputs), not raw dicts.
  2. **`IntradaySetupScanner`** — Does **not** fetch bars. Signature: `scan(bars_by_symbol: dict[str, list[Bar]], …)`. Handlers fetch bars then pass them in. To reuse intraday math inside something like a technical layer, import **`IntradayVWAPCalculator`**, **`IntradayEMA9Calculator`**, **`OpeningRangeBreakoutDetector`** (and related helpers in `day_trading_scanner.py`) — **not** the scanner class as a nested “analyzer.”
  3. **VIX** — Reuse `get_vix_snapshot_with_fallback` and tuple `VIX_SNAPSHOT_FALLBACK_SYMBOLS` from `stocvest/api/services/morning_brief_fetch.py` (`I:VIX` → `^VIX` → `VIX`). **Do not** add another VIX fetcher or a single hardcoded/configured VIX ticker as the primary approach; the **ordered fallback list** is the contract.
  4. **Day vs last trade scale** — `_parse_snapshot` already drops inconsistent `day` OHLC/VWAP when out of line with `lastTrade.p` (5× bound via `_session_day_prices_align_with_last`). Use **`snapshot.last_trade_price`** for current/near price; **do not** duplicate that sanity check downstream.
  5. **Extended hours** — `preMarket`/`premarket` and `afterHours`/`afterhours` key variants are normalized inside `_parse_snapshot` into `pre_market_*` / `after_hours_*` on **`Snapshot`**; use those fields, not raw dict access.

Signal **layer enums** and internal composite types are stable; **customer-facing** naming follows the Legal section above.
